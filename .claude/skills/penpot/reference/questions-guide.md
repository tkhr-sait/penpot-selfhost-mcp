# ヒアリングフレームワーク

ユーザーの要望が曖昧な場合に、効果的な質問で要件を引き出すためのガイド。

## 基本原則

- **一度に3つ以内**の質問にする
- **選択肢を提示**して回答しやすくする
- **AskUserQuestion ツール**を活用し、選択式で聞く
- 必須でない項目はデフォルト値で進める（完璧を求めて質問過多にしない）
- ユーザーが「おまかせ」と言ったら、デザイン原則に基づいて判断して進める

## シナリオ別テンプレート

### 1. 新規画面デザイン

**最初に聞くこと（必須）:**

```
Q1: どんな画面ですか？（目的と主要コンテンツ）
  例: ログイン画面、ダッシュボード、設定画面、LP

Q2: デバイスターゲットは？
  - デスクトップ (1280px) ← デフォルト
  - モバイル (375px)
  - タブレット (768px)
  - レスポンシブ（複数サイズ）

Q3: トンマナの方向性は？
  - クリーン＆ミニマル ← デフォルト
  - ダーク＆モダン
  - カラフル＆フレンドリー
  - プロフェッショナル＆フォーマル
```

**必要に応じて追加（2ラウンド目）:**
- 必要な要素・セクションの優先度
- 参考デザインやブランドガイドライン
- 特定のカラースキーム

### 2. コンポーネント作成

**最初に聞くこと:**

```
Q1: 何のコンポーネントですか？
  例: ボタン、カード、フォーム、ナビバー、モーダル

Q2: バリエーションは必要？
  - 単一バリエーションのみ
  - サイズ違い (S/M/L)
  - 状態違い (Default/Hover/Disabled)
  - タイプ違い (Primary/Secondary/Ghost)

Q3: 特定のスタイル要件は？
  - デフォルトで進める（セマンティックトークン使用）
  - カスタムカラー指定あり
  - 既存デザインに合わせる
```

### 3. 既存デザインの改修

**最初に聞くこと:**

```
Q1: どのデザインを変更しますか？
  - 現在選択中のシェイプ
  - ページ名/ボード名で指定
  - （具体的に教えてください）

Q2: どう変更しますか？
  - レイアウト変更
  - カラー/スタイル変更
  - 要素の追加/削除
  - テキスト内容の変更

Q3: 変更の範囲は？
  - 指定箇所のみ
  - ページ全体の統一
  - 全ページに適用
```

### 4. ワイヤーフレーム / 画面フロー

**最初に聞くこと:**

```
Q1: 何画面分のフローですか？
  - 単一画面
  - 2-3画面の小フロー
  - 5画面以上の大フロー

Q2: フィデリティレベルは？
  - Lo-Fi（グレーボックス、構造のみ）
  - Mid-Fi（基本スタイル付き）
  - Hi-Fi（完成度の高いデザイン）← デフォルト

Q3: 画面遷移の概要を教えてください
  例: ログイン → ダッシュボード → 詳細画面
```

## 判断基準: 質問すべきか即実行か

### 即実行してよいケース
- サイズ、色、フォントサイズなど具体的な数値指定がある
- 「〜のような」と参考が明示されている
- 修正指示が明確（「テキストを赤に」「パディングを16pxに」）
- コンポーネント名と仕様が明確

### ヒアリングが必要なケース
- 「いい感じに」「かっこよく」など主観的な表現
- 画面の目的や使用コンテキストが不明
- 複数の解釈が可能な指示
- 大規模なデザイン（5要素以上）で構成が不明

### デフォルト値（質問を省略する場合に使用）
- デバイス: デスクトップ (1280px幅)
- トンマナ: クリーン＆ミニマル
- フィデリティ: Hi-Fi
- 背景: surface-primary トークン
- テキスト: text-primary / text-heading トークン
- アクセント: accent-blue トークン
- スペーシング: 16px（標準）、24px（セクション間）
- 角丸: 8px（カード）、4px（入力）、24px（ピル型ボタン）
